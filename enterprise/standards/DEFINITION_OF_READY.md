# Definition of Ready (DoR)

> **Compliance References:**
> - Based on: Scrum Guide 2020
> - Spec: Product Backlog refinement
> - Controls: DoR checklist
> - See also: [governance/STANDARDS_COMPLIANCE_MATRIX.md](../STANDARDS_COMPLIANCE_MATRIX.md)

## Purpose
Minimum criteria a backlog item must meet before being taken into a sprint.

---

## User Story Level

For a user story to be READY:
- [ ] User story format is correct: "As a [persona], I want to [action], so that [value]"
- [ ] Acceptance criteria written (at least 3)
- [ ] Story points estimated (via Planning Poker)
- [ ] Max 8 SP (split if larger)
- [ ] Dependencies identified and resolved (or resolution plan exists)
- [ ] Required design/wireframe ready (for UI-related work)
- [ ] API contract defined (if backend work)
- [ ] Test scenarios drafted
- [ ] Technical questions answered (spike completed)
- [ ] Fits within sprint capacity

---

## Bug Fix Level

For a bug to be READY:
- [ ] Reproducible steps written
- [ ] Expected vs actual behavior is clear
- [ ] Affected environment/browser/device specified
- [ ] Severity determined (P1/P2/P3/P4)
- [ ] Screenshot or video attached
- [ ] Log/error message attached (if available)

---

## Technical Spike Level

For a spike to be READY:
- [ ] Question/hypothesis to investigate is clear
- [ ] Expected output defined (report, PoC, decision)
- [ ] Time-boxed (max 2 days)
- [ ] Success criteria defined

---

## IF NOT READY
- CANNOT be taken into the sprint
- Returns to the Product Owner
- Required information is completed
- Re-evaluated in the next grooming session
