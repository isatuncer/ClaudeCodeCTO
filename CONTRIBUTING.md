## Ways to Contribute

You can help by:

- **Reporting bugs** — see [Reporting Bugs](#reporting-bugs)
- **Suggesting features** — see [Suggesting Features](#suggesting-features)
- **Improving translations** — the README exists in 10 languages at [`docs/i18n/`](docs/i18n/)
- **Proposing new source repositories** — open an issue arguing why a new repo should be added to `.gitmodules`
- **Fixing installer bugs** — the install flow is in `scripts/`
- **Improving documentation** — README, CHANGELOG, inline comments

**What NOT to contribute:**
- ❌ Manual edits to `decisions/selected.json` — this file is regenerated by the curation pipeline. Instead, open an issue explaining why a component should be added, removed, or re-scored.
- ❌ New skills/agents/commands as standalone files — add them to one of the upstream repos in `.gitmodules` instead, and they'll flow into the next curation run.

## Reporting Bugs

Before opening a bug report, please:

1. Check [existing issues](https://github.com/isatuncer/ClaudeCodeCTO/issues) to see if it's already reported
2. Verify the bug with a clean install: `rm -rf ~/.claude && bash scripts/setup.sh`
3. Collect the following:
   - OS and version (Windows 11, macOS 14, Ubuntu 22.04, etc.)
   - Output of `bash --version`, `python --version`, `git --version`
   - Contents of `decisions/setup.log` (generated by `setup.sh`)
   - Contents of `decisions/smoke-test-report.md` if the smoke test failed

Open an issue using the **Bug Report** template. Include:

- A clear description of what went wrong
- Steps to reproduce
- Expected vs. actual behavior
- The log snippets above
- Screenshots if the issue is visual

## Suggesting Features

Open an issue using the **Feature Request** template. Explain:

- **What** — describe the feature in one sentence
- **Why** — what problem does it solve? Who benefits?
- **How** — rough sketch of implementation, if you have one
- **Alternatives** — what other approaches did you consider?

Feature requests that touch the **curation pipeline** should explain why a new scoring dimension or domain is needed, not just "I want skill X included" — for individual component inclusion, open an issue about the component directly.

## Improving Translations

Translations live at [`docs/i18n/README.<lang>.md`](docs/i18n/). Each mirrors the structure of the main [`README.md`](README.md) but is condensed (~300–400 lines).

To fix or update a translation:

1. Fork the repo
2. Edit the relevant `docs/i18n/README.<lang>.md` file
3. Keep the language switcher at the top in sync (list of all 10 languages)
4. Open a PR with the prefix `docs(i18n):` in the commit message

To add a new language:

1. Copy `docs/i18n/README.de.md` as a starting template
2. Translate each section — keep headings and tables consistent with other languages
3. Add the new language to the switcher in **all** existing README files (root `README.md`, `README.tr.md`, and all `docs/i18n/*.md`)
4. Open a PR

## Submitting Code Changes

1. **Fork** the repository and create a feature branch: `git checkout -b feat/short-description`
2. **Make your changes** — keep commits focused and atomic
3. **Test locally:**
   - `bash scripts/setup.sh --dry-run` — verify no errors
   - `bash scripts/setup.sh --check` — verify state inspection works
   - For changes to `installer.sh`: run a full dry-run install
4. **Run the smoke test:** `bash scripts/smoke_test.sh`
5. **Open a pull request** with a clear title and description following the [commit message style](#commit-message-style)
6. **Respond to review feedback** — maintainers may request changes

### Pull Request Requirements

- ✅ PR title follows [Conventional Commits](https://www.conventionalcommits.org/) format
- ✅ Description explains the **why**, not just the what
- ✅ Links to the relevant issue (if applicable)
- ✅ No changes to `decisions/selected.json` unless you're the maintainer
- ✅ No secrets, credentials, or user-specific paths committed
- ✅ CI checks pass (shellcheck, JSON validation, smoke test dry-run)

## Development Setup

```bash
git clone https://github.com/isatuncer/ClaudeCodeCTO.git
cd ClaudeCodeCTO
git submodule update --init --recursive
python -m pip install pyyaml
bash scripts/setup.sh --check  # verify without installing
```

The install flow (5 scripts) is on GitHub. The analysis pipeline (8 Python scripts) is **local-only on the maintainer's machine** and not pushed — see [`.gitignore`](.gitignore). Contributors working on installer or template changes don't need the pipeline.

## Repository Layout

```
ClaudeCodeCTO/
├── install.sh                  ← curl-pipe compatible one-command installer
├── scripts/                    ← install infrastructure (GitHub-tracked)
│   ├── setup.sh                main entry point (6 phases)
│   ├── bootstrap.sh            first-time clone wrapper
│   ├── installer.sh            atomic staged install with backup
│   ├── smoke_test.sh           post-install verification (8 tests)
│   └── tracker.sh              optional usage tracking
├── decisions/                  ← single source of truth (pre-built curation)
│   ├── selected.json           the 2,388-component authoritative list
│   ├── install-assets.json     embedded orchestrator files
│   ├── install-manifest.json   last install checkpoint
│   ├── lifecycle-bindings.json 8-phase → component mapping
│   ├── budget-profile.json     token cost profile
│   └── agent-decision-tree.md  agent disambiguation tree
├── sources/                    ← 15 git submodules (actual content)
├── docs/i18n/                  ← translated READMEs (10 languages)
└── .github/                    ← community files + CI
```

## Commit Message Style

We use [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>(<scope>): <short summary>

<optional body explaining the why>

<optional footer with breaking changes or issue refs>
```

**Types:**
- `feat` — new feature
- `fix` — bug fix
- `docs` — documentation only
- `refactor` — code change that neither fixes a bug nor adds a feature
- `perf` — performance improvement
- `test` — adding/updating tests
- `chore` — tooling, dependencies, repo maintenance
- `ci` — CI/CD changes

**Scopes:** `installer`, `setup`, `smoke`, `curation`, `i18n`, `docs`, etc.

**Examples:**
```
feat(installer): add --dry-run flag for safe previews
fix(smoke): handle missing CLAUDE.md gracefully
docs(i18n): fix Spanish quick-start example
refactor: embed orchestrator files into install-assets.json
```
